# Plant Profile + Dashboard — Plan (software track)

> **You may want to start at [`plan.md`](plan.md) instead.** That's the master plan and entry point for the project — it has the current status snapshot across both tracks and links here for software details. This doc is **track B (software)**: per-plant profiles, the data pipeline, the dashboard, and the public deployment. The hardware track (sensor → relay → pump → field test) is owned by `plan.md`.

Goal: a per-plant profile + dashboard that lets Winnie (and Xiaoxia) actually *see* what's going on with each plant.

---

## Goal

Give every plant in the household a profile page that answers four questions at a glance:

1. **Who are you?** — name, species, photo (with the date it was taken).
2. **How should I take care of you?** — watering / sunlight / soil / temperature tips, generated by Claude from the species name and editable.
3. **How are you doing right now?** — most recent moisture reading and the time it was taken.
4. **What's your watering history?** — the last ~3 months of waterings with before-and-after moisture %.

Two surfaces share the same underlying data: **Telegram** (Xiaoxia, for quick "how is Bertha doing?" checks on the phone) and a **minimal per-plant web page** on the VPS, regenerated each time a new reading lands and accessed via the same SSH-tunnel pattern as the existing OpenClaw dashboard.

## Architecture

```
Arduino Uno (≥1 moisture sensor + ≥1 relay)
   │  Serial CSV @ 9600 baud  (multi-plant format — see below)
   ▼
Mac /dev/cu.usbmodem1301
   │  socat TCP:12345  →  SSH reverse tunnel  (already in arduino-serial-bridge.md)
   ▼
VPS (clawd@2.24.221.71)
   ├── waterplant-logger (new daemon)
   │     reads TCP stream, attaches host-side timestamp, writes to SQLite
   ├── SQLite DB        /var/lib/waterplant/waterplant.db
   ├── Photo store      /var/lib/waterplant/photos/<plant_id>/<iso_ts>.jpg
   ├── Static site gen  /var/www/waterplant/<plant_id>.html  (regenerated each reading)
   ├── HTTP server      nginx or python -m http.server, behind an SSH tunnel
   └── Xiaoxia (LLM agent in VM)
         reads the serial stream and operates this repo directly:
         commits YAML edits, runs render.py, regenerates pages
```

The Arduino stays a dumb sensor + relay driver. Everything stateful splits cleanly in two: **time-series data** (every moisture reading, every watering) lives in SQLite where the logger writes it, and **plant metadata** (name, species, care tips, notes, photo manifest) lives as YAML files inside this repo, which Xiaoxia edits and commits. Photos themselves stay on the VPS filesystem because they're too big for git, but each photo is referenced from its plant's YAML so the manifest is versioned.

This split is deliberate: Xiaoxia is an LLM agent with codebase access, so the "human" data lives where she's most fluent — text files in git. The "machine" data stays in a database where it belongs.

## Required Arduino changes

The current CSV is `millis,moisture_pct,event` with no plant identifier — fine when there's exactly one sensor, but breaks the moment a second plant joins. The smallest change that buys multi-plant support is to add a sensor index:

```
millis,sensor_idx,moisture_pct,event
```

`sensor_idx` is a small integer (0 = A0, 1 = A1, …). The mapping `sensor_idx → plant_id` lives on the host, not the Arduino, so we never need to reflash the firmware just because a plant got renamed.

Concretely, `config.h` grows a `SENSORS[]` array:

```cpp
struct SensorPump {
  uint8_t moisturePin;   // e.g. A0
  uint8_t relayPin;      // e.g. 7  (or 0xFF if this plant has no pump)
  int     sensorDry;     // calibrated per sensor
  int     sensorWet;
};

static const SensorPump SENSORS[] = {
  { A0, 7, 458, 265 },   // sensor_idx=0
  // { A1, 8, 470, 270 }, // sensor_idx=1 — uncomment when adding plant 2
};
```

`WaterPlant.ino`'s loop iterates `SENSORS[]`, reads each one in turn, and emits one CSV row per sensor per `READ_INTERVAL_MS`. The pump safety logic (cooldown, hard cutoff) is enforced per-sensor.

### A note on "moisture before vs. after watering"

The current sketch logs `PUMP_OFF` with the same `pct` it logged at `PUMP_ON` (no re-read). For the dashboard we want a real "after" value. The cleanest path is to skip a firmware change entirely: **the next periodic reading (60 s after the pump turns off) is the "after" value**. Soil takes time to redistribute moisture anyway — an immediate post-pump read is misleadingly low. The host-side logger pairs each `PUMP_OFF` row with the next regular reading from the same `sensor_idx` and stores both.

If we ever want a tighter loop (e.g. 30 s settle + dedicated post-water read), we add a `POST_WATER` event in the firmware later. Not needed for v1.

## Data layout

Two stores, one for each kind of state.

### Time-series in SQLite

A single SQLite file on the VPS — no network DB, no migrations to worry about. Two tables, both written by the logger daemon:

```sql
-- Every moisture reading the logger ingests.
CREATE TABLE readings (
  id              INTEGER PRIMARY KEY,
  plant_id        INTEGER NOT NULL,        -- resolved from sensor_idx via the repo's plants/*.yaml
  recorded_at     TEXT NOT NULL,           -- host wall-clock at receive (ISO 8601)
  arduino_millis  INTEGER NOT NULL,        -- raw uptime from Arduino, debug aid only
  moisture_pct    INTEGER NOT NULL
);
CREATE INDEX idx_readings_plant_time ON readings(plant_id, recorded_at);

-- One row per watering. moisture_after is filled in async by the logger
-- the next time it sees a regular reading from the same plant.
CREATE TABLE watering_events (
  id                INTEGER PRIMARY KEY,
  plant_id          INTEGER NOT NULL,
  started_at        TEXT NOT NULL,
  ended_at          TEXT,
  moisture_before   INTEGER,                -- last reading before PUMP_ON
  moisture_after    INTEGER,                -- next reading after PUMP_OFF
  duration_ms       INTEGER,
  trigger_source    TEXT NOT NULL DEFAULT 'auto'   -- 'auto' | 'manual'
);
```

Backups are a nightly `cp` of the `.db` file plus `rsync` of the photos directory off the VPS. Trivial because everything is files.

### Metadata in YAML (versioned in git)

Each plant gets one YAML file at `plants/<id>.yaml`. This is what Xiaoxia edits and commits when the user adds a plant, uploads a photo, or jots a note. Single file per plant means clean git diffs and no merge conflicts when editing different plants in parallel.

```yaml
# plants/1.yaml
id: 1
name: Bertha
species: Monstera deliciosa
sensor_idx: 0           # maps to SENSORS[0] on the Arduino
pump_idx: 0
added_at: 2026-04-27
care_tips:              # generated once at /addplant time, edited freely
  watering: "Water when the top 2 in of soil is dry — usually weekly in summer..."
  sunlight: "Bright, indirect light. Tolerates lower light but grows slower."
  soil: "Well-draining mix: peat, perlite, and bark."
  temperature: "65–85°F. Avoid cold drafts and dry heating vents."
photos:                 # one entry per upload; file lives on VPS, not in git
  - path: 1/2026-04-24.jpg
    taken_at: 2026-04-24
    caption: "Repotted into 8 in terracotta"
    is_hero: true
notes:                  # chronological, newest first
  - date: 2026-04-24
    body: "Repotted into 8 in terracotta with fresh peat/perlite mix."
  - date: 2026-04-10
    body: "New leaf unfurled — fenestrations getting bigger."
```

A small `plants/_sensor_map.yaml` (auto-generated from the individual plant YAMLs by a pre-commit hook or `make` target) gives the logger a quick `sensor_idx → plant_id` lookup without parsing every plant file at startup.

## What's on a plant profile page

Each plant gets one page at `/<plant_id>.html`. Sections, top to bottom:

| Section | Source | Updates when |
|---|---|---|
| Header (name, species, hero photo, "last seen N min ago") | plants + most recent photo + most recent reading | every reading |
| Right-now card (current moisture %, threshold band, last watered, predicted next watering) | readings + watering_events | every reading |
| Care tips (watering / sunlight / soil / temperature) | plants.care_tips_json | when user edits via Telegram or web |
| Moisture chart (last 30 days, threshold shaded) | readings | every reading |
| Watering history (last 3 months: date, before %, after %, duration, trigger) | watering_events | every watering |
| Photo timeline (chronological strip with dates) | photos | every upload |
| Notes log (chronological) | notes | every note |
| Sensor health ("last reading 2 min ago" / "⚠ no data for 1 h") | derived from readings | every reading |

The page is plain HTML + CSS + a single `<script>` block that draws the moisture chart from JSON embedded by the generator. No build step, no SPA framework — generation cost stays in the millisecond range.

### "Anything else?" — additions beyond your four bullets

Worth including in v1 because they're cheap once the data is already there:

- **Sensor health indicator.** A flatlined or missing reading should look obviously broken on the page. Easy: derive from `MAX(recorded_at)` per plant.
- **Predicted next watering.** Linear fit on the dry-down rate over the last few days → "expected to hit 30% on Sat". Cheap heuristic, surprisingly useful.
- **Pump-effectiveness flag.** If `moisture_after - moisture_before < 5` for two consecutive waterings, flag "pump may be empty / tube kinked".
- **Notes log.** Catches the things sensors can't see: fertilized, repotted, yellow leaf, treated for pests. Same table is reused for any future event types.
- **Photo timeline as a strip, not just one hero shot.** You already have the date for each photo — let the user scroll through growth over time.

Worth deferring (not v1):

- **Manual "water now" from Telegram.** Requires the Arduino to *read* serial input, which it currently doesn't. Add in a later phase as a serial command parser (`WATER 0\n` → trigger pump 0). Same code path could later accept threshold updates.
- **Light / temperature / humidity sensors.** New hardware. Schema is friendly to it (add a `metric` column to `readings` or a sibling table).
- **Multi-user / sharing.** YAGNI for one household.

## How Xiaoxia operates the system

Xiaoxia is an LLM agent in a VM with two relevant capabilities: she can read the live serial stream from the bridge, and she can edit + commit this repo on GitHub. She doesn't run a Telegram-style command parser — she's instructed in natural language and does the right thing.

The five operations Xiaoxia is expected to handle:

| Ask | What Xiaoxia does |
|---|---|
| "Add Bertha as a Monstera deliciosa" | Picks the next free `id`, creates `plants/<id>.yaml` with name + species + sensor_idx, calls the care-tip generator (see prompt below), fills `care_tips`, regenerates `_sensor_map.yaml`, runs `render.py <id>`, commits with message `add plant: Bertha (Monstera deliciosa)`. |
| "Here's a photo of Bertha from today" | Saves the image to `/var/lib/waterplant/photos/<id>/<iso_date>.jpg` on the VPS, appends a `photos:` entry in the YAML, runs `render.py <id>`, commits the YAML change. |
| "How's Bertha?" | Reads the latest `readings` row for her sensor + recent `watering_events` directly from SQLite, summarizes in chat, and links to the rendered page. No commit. |
| "Bertha just got fertilized" | Appends a `notes:` entry with today's date, runs `render.py`, commits. |
| "Update Bertha's watering tip — she's actually fine with bi-weekly" | Edits `care_tips.watering` in the YAML, runs `render.py`, commits with message `edit care tips: Bertha`. |

Two things keep this clean: (1) Xiaoxia only writes to YAML files in git, never to SQLite — that's the logger's lane, so we never have two writers fighting over the same rows; (2) every metadata change goes through a commit, so the project history is also a complete audit log of "what did Xiaoxia do to my plants."

### Rules of the road for Xiaoxia

These exist because of real misfires we want to avoid repeating. Each rule has a one-line *why* so it can be edited later as the system evolves.

- **Never replace existing files with `# Dummy ... placeholder` stubs.** Files in `WaterPlant/`, `calibrate/`, `tests/`, `lesson/`, `docs/`, and `dashboard_preview/` are real production artifacts. *Why:* on 2026-04-27 the firmware, the calibration sketch, the Phase 1 test writeup, and the styled dashboard preview all got replaced with stubs in a single commit, blowing away calibrated values (`SENSOR_DRY=458`, `SENSOR_WET=265`) and 480 lines of styled HTML. Recovery cost a corrective commit.
- **Never commit `server/.venv/` or any local virtualenv.** That directory is in `.gitignore`. *Why:* venvs contain ~1,000 binary files that bloat the repo, leak the runner's Python version, and make diffs unreadable.
- **Edit plant metadata only in `plants/<id>.yaml`.** Never write to `plants/_sensor_map.yaml` by hand — regenerate it via `python server/render.py --rebuild-sensor-map`. *Why:* the map is derived from the individual plant files; hand-edits will drift and the next regeneration will silently overwrite them.
- **Time-series data is the logger's lane.** Don't insert into `readings` or `watering_events` from scripts other than `server/logger.py`. *Why:* two writers on the same SQLite tables risks lost-update races and confuses the audit trail.
- **Real comments only in real code.** If a file has actual content, it does NOT get a `# Dummy ... placeholder` comment at the top. *Why:* labeling a real schema as "Dummy" makes future readers (human and LLM) treat it as throwaway and risk replacing it.
- **The styled `dashboard_preview/1.html` is production for the public site.** If the styled layout needs to change, edit the HTML directly or the Jinja template at `server/templates/plant.html.j2` — but keep them in visual sync. *Why:* `dashboard_preview/` is what GitHub Pages serves to anyone with the URL.

### The care-tip prompt template

Care tips are generated once when a plant is added, then edited freely later. The "prompt template" is just a single file (`prompts/care_tips.md`) that pins the system prompt + a worked example Xiaoxia uses every time she generates tips. **It exists so two Monsteras get tips in the same shape**: same four fields (`watering`, `sunlight`, `soil`, `temperature`), same length, same tone. Without a pinned template, asking on Tuesday vs. Friday could yield differently-structured outputs and the page renderer (which expects exactly those four keys) would break.

Concretely, the file looks something like:

```
# prompts/care_tips.md

You are filling in the care_tips block for a new plant. Output **YAML only** —
exactly four keys: watering, sunlight, soil, temperature. Each value is one
or two short sentences (~20–40 words). Practical, concrete, no marketing fluff.

# Example for Monstera deliciosa:
care_tips:
  watering: "Water when the top 2 in of soil is dry — usually weekly in summer, every 10–14 days in winter."
  sunlight: "Bright, indirect light. Tolerates lower light but grows slower and produces fewer fenestrations."
  soil: "Well-draining mix: peat, perlite, and bark. Repot every 2–3 years."
  temperature: "65–85°F. Avoid cold drafts and dry heating vents."

# Now write care_tips for: {{species}}
```

If we ever want to tighten or loosen the format (add a `fertilizer` field, drop `temperature`), we change one file and every future plant's tips follow.

#### Plant-sprite prompt template — pinned at [`prompts/sprites.md`](../prompts/sprites.md)

The full style guide, the "preserve plant / reinvent pot" rule, the prompt template with `{{ }}` placeholders, the worked Goldie example, and the current invocation pattern all live in [`prompts/sprites.md`](../prompts/sprites.md). Per-plant inputs (which features to preserve, which artistic pot to use, where the reference photo lives) are stored in each plant's `sprite:` block in `plants/<id>.yaml` — see [`plants/1.yaml`](../plants/1.yaml) for the canonical example.

The most important convention from that file, repeated here so it doesn't get lost:

> **PRESERVE the plant, REINVENT the pot.** Cartoon avatars should faithfully copy the photo's leaf shape, variegation, growth habit, and proportions, but the pot is artistic license — never copy the real plastic/utilitarian pot from the photo. Use a hand-woven wicker basket, hand-painted ceramic, or similar.

#### Sprite generation — Gemini 2.5 Flash Image (nanobanana) — deferred to D7c

All sprites are generated via the Generative Language API model `gemini-2.5-flash-image`. Prompts live in `prompts/sprites.md` (to be added in this phase) so the entire library is reproducible — if we redesign Xiaoxia or repaint Goldie, we re-run the same prompts and commit the new PNGs.

API key is loaded from a gitignored `.env` at the repo root (`GEMINI_API_KEY=...`). Generation is wrapped in `server/gen_sprites.py <name>`, which reads the prompt for `<name>` from `prompts/sprites.md`, calls the API, and saves the PNG under `dashboard_preview/sprites/<name>.png`. Idempotent — re-running overwrites the same file.

## Risks / things to think through before D1

A few traps worth flagging upfront so we don't trip over them mid-build:

The Arduino has no RTC, only `millis()`, which resets to 0 on every boot. This is fine — the logger uses host wall-clock — but it means `arduino_millis` alone is **not** a usable timestamp; it's just an "uptime" debug aid. The schema reflects this (the timestamp column is `recorded_at`, set host-side).

The serial bridge currently flows Mac→VPS one-way. If the Mac is asleep or unplugged, the Arduino keeps running but nothing on the VPS sees it. The logger should treat "no data for N minutes" as a sensor-health red flag, not as "moisture is fine because the last reading was good." This is what the sensor-health indicator in D5 is for.

The serial-bridge doc and the actual sketch disagree on baud rate (`115200` vs `9600`). Worth fixing before D1 — picking 9600 to match the calibration sketch and leaving it there.

Photo storage is unbounded. A Pi-style 16GB VPS disk fills surprisingly fast if you take a daily photo per plant for a year. Easy mitigation: cap each plant at, say, 365 photos (oldest auto-archived to Mac via the nightly backup).

### 2026-04-27 (visual polish — D7 design)

9. **Index becomes a pixel garden, not a card grid.** The current `dashboard_preview/index.html` (cards-style) is replaced by an open pixel-art scene. Per-plant detail pages (`<id>.html`) keep their existing chart/history layout. *Why:* the cards version is functional but generic; the garden gives the project an identity (Xiaoxia as a character, plants as avatars) and a hook for event-driven animation later.
10. **Layout: open garden.** All plants share one scene; no rooms grid. *Why:* matches the inspiration screenshot's vibe, scales to ~10 plants in a single household, and gives Xiaoxia somewhere to walk in D7b.
11. **Xiaoxia is a literal small lobster.** Pink/red shell, two claws. *Why:* matches the playful name, distinctive against green plants, makes "Xiaoxia is watering Bertha" a visually memorable moment in D7b.
12. **Health indicators are floating icons, not pose changes or color tints.** 💧 thirsty / ✨ just-watered / ⚠ sensor offline / nothing = healthy. *Why:* keeps v1 cheap (one sprite per plant, not multiple poses) and the icons read at a glance even if you're not familiar with the plant.
13. **Animation tier: static first, event-driven later (D7b).** v1 ships zero motion in the garden; movement waits for real sensor data. *Why:* avoids burning time on walk cycles and watering animations before we know the data wiring works, and lets us ship a visible improvement now.
14. **No mobile fallback in D7 v1.** Garden is desktop/tablet only. *Why:* fitting an open garden into a 360 px wide column is a separate design problem; punting keeps this phase scoped.
15. **Sprite source: Gemini 2.5 Flash Image (nanobanana) via API.** Prompts pinned in `prompts/sprites.md`; generation script at `server/gen_sprites.py`; PNGs committed under `dashboard_preview/sprites/`. *Why:* reproducible (re-run a prompt → same kind of sprite), version-controlled prompts give us a record of every asset's intent, and the API is cheap enough (~$1.20 for the full v1 library) that we can iterate on prompts freely.
